# Resource Consumption

### 1. การเตรียมข้อมูลจำลอง (Dummy Data)

```
CREATE TABLE test_work_mem AS
SELECT
id,
md5(random()::text) AS sort_key, -- ข้อความสุ่มยาวๆ เพื่อให้เปลืองพื้นที่ในการ Sort
random() * 100 AS val
FROM generate_series(1, 1000000) id; -- 1 ล้านแถว
```

### 2. วิธีทดสอบ: สังเกตการใช้ Disk (External Merge)

```
-- ลองตั้งค่า work_mem ให้ต่ำๆ เพื่อดูผลลัพธ์แบบที่ใช้ Disk
SET work_mem = '4MB';

EXPLAIN ANALYZE
SELECT * FROM test_work_mem ORDER BY sort_key;
```

สิ่งที่ต้องมองหาในผลลัพธ์:

- ถ้าเห็นคำว่า Sort Method: external merge Disk: XXXkB: แสดงว่า work_mem ไม่พอ ข้อมูลต้องถูกเขียนลง Disk (ช้า)
- ถ้าเห็นคำว่า Sort Method: quicksort Memory: XXXkB: แสดงว่า work_mem เพียงพอ ข้อมูลจัดการใน RAM ทั้งหมด (เร็ว)

### 3. การทดสอบด้วย Hash Join (สำหรับ Join ตารางใหญ่)

work_mem ไม่ได้ใช้แค่การ Sort แต่ใช้กับการสร้าง Hash Table เวลา Join ด้วย

```
SET work_mem = '4MB';

EXPLAIN ANALYZE
SELECT count(*)
FROM test_work_mem t1
JOIN test_work_mem t2 ON t1.sort_key = t2.sort_key;
```

สิ่งที่ต้องมองหา:

- มองหาคำว่า Buckets: XXX Batches: Y Memory Usage: ZZZkB
- ถ้า Batches มากกว่า 1 แสดงว่ามีการแบ่งข้อมูลไปเขียนที่ Disk เพราะ RAM ไม่พอสำหรับสร้าง Hash Table ในครั้งเดียว

### 4. SQL ตรวจสอบการเกิด Temporary Files (ย้อนหลัง)

หากอยากรู้ว่าที่ผ่านมามีการใช้ Disk แทน Memory บ่อยแค่ไหน ให้ใช้คำสั่งนี้เช็คจากระบบ:

```
SELECT
    datname,
    temp_files AS count,
    pg_size_pretty(temp_bytes) AS total_size
FROM pg_stat_database
WHERE datname = current_database();
```
