# Shutting Down the Server

มีหลายวิธีในการปิดเซิร์ฟเวอร์ฐานข้อมูล โดยพื้นฐานแล้ว ทุกวิธีล้วนเป็นการส่งสัญญาณไปยังกระบวนการควบคุม postgres

สามารถควบคุมประเภทของการปิดระบบได้โดยการส่งสัญญาณต่างๆ ไปยังกระบวนการ postgres:

- **_SIGTERM_**
  `Smart Shutdown mode` หลังจากได้รับสัญญาณ _SIGTERM_ เซิร์ฟเวอร์จะไม่อนุญาตให้มีการเชื่อมต่อใหม่ แต่จะอนุญาตให้เซสชันที่มีอยู่ ทำงานจนเสร็จสิ้นตามปกติ เซิร์ฟเวอร์จะปิดระบบก็ต่อเมื่อ เซสชันทั้งหมดสิ้นสุดลงแล้วเท่านั้น หากเซิร์ฟเวอร์อยู่ `recovery mode` เมื่อมีการร้องขอ `Smart Shutdown mode` การ `recovery` และ `streaming replication` จะหยุดลงก็ต่อเมื่อ เซสชันปกติทั้งหมดสิ้นสุดลงแล้วเท่านั้น
- **_SIGINT_**
  `Shutdown mode` เซิร์ฟเวอร์จะไม่อนุญาตให้มีการเชื่อมต่อใหม่ และส่งสัญญาณ SIGTERM ไปยังกระบวนการเซิร์ฟเวอร์ที่มีอยู่ทั้งหมด ซึ่งจะทำให้กระบวนการเหล่านั้นยกเลิกธุรกรรมปัจจุบัน และหลุดออกทันที จากนั้นรอจนกว่ากระบวนการเซิร์ฟเวอร์ทั้งหมดจะปิดตัวลง
- **_SIGQUIT_**
  `Immediate Shutdown mode` เซิร์ฟเวอร์จะส่งสัญญาณ SIGQUIT ไปยังกระบวนการลูกทั้งหมด และรอจนกว่ากระบวนการเหล่านั้นจะสิ้นสุดการทำงาน. หากกระบวนการใดไม่สิ้นสุดการทำงานภายใน 5 วินาที เซิร์ฟเวอร์จะส่งสัญญาณ SIGKILL ไปยังกระบวนการเหล่านั้น. กระบวนการเซิร์ฟเวอร์ผู้ควบคุม จะหลุดออกจากระบบหลังจาก ที่กระบวนการลูกทั้งหมดสิ้นสุดการทำงาน, โดยไม่ต้องดำเนินการปิดระบบฐานข้อมูลตามปกติ. ซึ่งจะนำไปสู่การกู้คืน (โดยการเล่นบันทึก WAL [^1] ซ้ำ) ในการเริ่มต้นระบบครั้งถัดไป แนะนำให้ใช้โหมดนี้เฉพาะในกรณีฉุกเฉินเท่านั้น.

โปรแกรม `pg_ctl` มีอินเทอร์เฟซที่สะดวกสำหรับการส่งสัญญาณเหล่านี้เพื่อปิดเซิร์ฟเวอร์. หรืออีกวิธีหนึ่ง สามารถส่งสัญญาณโดยตรงโดยใช้คำสั่ง kill บนระบบที่ไม่ใช่ Windows. PID ของ postgres process สามารถพบได้ โดยใช้โปรแกรม ps, หรือจากไฟล์ postmaster.pid ใน `data directory`.

ตัวอย่างในการปิดระบบอย่างรวดเร็ว:

```
ubuntu:~$ sudo kill -s 15 -INT $(sudo head -1 /usr/local/pgsql-17.9/data/postmaster.pid)
```

## Important

ดีที่สุดอย่าใช้ SIGKILL เพื่อปิดเซิร์ฟเวอร์. การทำเช่นนั้นจะทำให้เซิร์ฟเวอร์ ไม่สามารถปลดปล่อย `shared memory` และ `semaphores` คืนได้. นอกจากนี้ SIGKILL จะฆ่า `postgres process` โดยไม่ถ่ายทอดสัญญาณต่อไปยัง `subprocesses`, ดังนั้นอาจจำเป็นต้องฆ่า `subprocesses` แต่ละรายการด้วยตนเองเช่นกัน.

ในการยุติเซสชันเฉพาะบุคคล ในขณะอนุญาตให้เซสชันอื่นทำงานต่อไป, ใช้ pg_terminate_backend() หรือส่งสัญญาณ SIGTERM ไปยังกระบวนการลูก ที่เกี่ยวข้องกับเซสชันนั้น.
