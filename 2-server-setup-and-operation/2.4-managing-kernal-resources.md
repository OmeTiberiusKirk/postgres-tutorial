# Managing Kernel Resources

บางครั้ง PostgreSQL อาจใช้ทรัพยากรต่างๆ ของ OS จนหมด โดยเฉพาะเมื่อมีการใช้งานหลาย `instance บนระบบเดียวกัน หรือ instance มีขนาดใหญ่มาก ส่วนนี้จะอธิบายถึง kernel resource ที่ PostgreSQL ใช้ และขั้นตอนที่คุณสามารถทำได้ เพื่อแก้ไขปัญหาที่เกี่ยวข้องกับการใช้ kernel resource

## Shared Memory and Semaphores

PostgreSQL ต้องการให้ OS มีคุณสมบัติการสื่อสารระหว่างกระบวนการ (IPC) โดยเฉพาะอย่างยิ่ง shared memory [^1] และ semaphores [^2] ระบบที่พัฒนามาจาก Unix โดยทั่วไปจะมี IPC แบบ System V [^3] (เก่า) หรือ POSIX [^4] (ใหม่) หรือทั้งสองแบบ

นอกจากนี้ ยังมีการสร้าง semaphores ซึ่งอาจเป็นแบบ System V หรือ POSIX ในระหว่างการเริ่มต้นเซิร์ฟเวอร์ ปัจจุบัน semaphores แบบ POSIX ถูกใช้ในระบบ Linux และ FreeBSD ในขณะที่แพลตฟอร์มอื่นๆ ใช้ semaphores แบบ System V

```
# System V
ipcs -m

# POSIX
ls -lah /dev/shm
```

คุณสมบัติ IPC ของ System V มักถูกจำกัดด้วยขีดจำกัดการจัดสรรทั่วทั้งระบบ เมื่อ PostgreSQL เกินขีดจำกัดเหล่านี้ เซิร์ฟเวอร์จะปฏิเสธการเริ่มต้น และจะทิ้งข้อความแสดงข้อผิดพลาด ที่ให้คำแนะนำไว้

```
postgres:~$ ipcs -lms
------ Shared Memory Limits --------
max number of segments = 4096 # SHMMNI
max seg size (kbytes) = 18014398509465599 # SHMMAX (หน่วยเป็น KB)
max total shared memory (kbytes) = 18446744073709551612 # SHMALL (หน่วยเป็น KB)
min seg size (bytes) = 1  # SHMMIN

------ Semaphore Limits --------
max number of arrays = 32000
max semaphores per array = 32000
max semaphores system wide = 1024000000
max ops per semop call = 500
semaphore max value = 32767
```

## systemd RemoveIPC

หากมีการใช้งาน `systemd` ต้องระมัดระวังในการใช้ทรัพยากร `IPC` (รวมถึง `shared memory`) ถูก `OS` ลบทิ้งก่อนเวลาอันควร โดยเฉพาะอย่างยิ่งเมื่อติดตั้ง PostgreSQL จากซอร์สโค้ด เพราะการติดตั้งแบบอื่น ผู้ใช้ `postgres` จะถูกสร้างขึ้นเป็น `system user`

การตั้งค่า RemoveIPC จะอยู่ในในไฟล์ `/etc/systemd/logind.conf` จะควบคุมว่า IPC จะถูกลบออกหรือไม่ เมื่อผู้ใช้ล็อกเอาต์อย่างสมบูรณ์ (ยกเว้น `system user`) การตั้งค่านี้มีค่าเริ่มต้นเป็น `เปิด` ใน `systemd` มาตรฐาน แต่ระบบปฏิบัติการบางรุ่นอาจมีค่าเริ่มต้นเป็นปิด

ผลกระทบที่พบได้ทั่วไป เมื่อเปิดใช้งานการตั้งค่านี้คือ `shared memory objects` ซึ่งใช้สำหรับ `Parallel Query Execution` จะถูกลบออกในเวลาที่ไม่แน่นอน ส่งผลให้เกิดข้อผิดพลาด เช่น

```
WARNING:  could not remove shared memory segment "/PostgreSQL.1450751626": No such file or directory
```

## Resource Limits

ระบบปฏิบัติการแบบ Unix มีข้อจำกัดด้านทรัพยากรหลายประเภท ที่อาจรบกวนการทำงานของเซิร์ฟเวอร์ PostgreSQL

ข้อจำกัดที่สำคัญอย่างยิ่ง ได้แก่ จำนวนกระบวนการต่อผู้ใช้ จำนวนไฟล์ที่เปิดได้ต่อกระบวนการ และปริมาณหน่วยความจำที่ใช้ได้สำหรับแต่ละกระบวนการ แต่ละข้อจำกัดเหล่านี้มี "ขีดจำกัดแบบตายตัว" และ "ขีดจำกัดแบบยืดหยุ่น"

### Footnotes

[^1]:
    Shared Memory คือ หน่วยความจำส่วนกลางที่ถูกจัดสรรไว้ให้ Process หลายๆ ตัวสามารถเข้าถึงและใช้งานร่วมกันได้ โดยปกติแล้วในระบบปฏิบัติการ Linux/Unix แต่ละ Process จะมีพื้นที่หน่วยความจำของตัวเอง (Private Memory) ซึ่งตัวอื่นจะเข้าไปยุ่งไม่ได้ เพื่อความปลอดภัย <br><br>
    แต่สำหรับ Database อย่าง PostgreSQL ที่มีการทำงานแบบ Multi-process (มี Process แม่และ Process ลูกหลายตัวช่วยกันทำงาน) จำเป็นต้องมีพื้นที่ส่วนกลางนี้เพื่อแลกเปลี่ยนข้อมูลกันอย่างรวดเร็ว

[^2]: Semaphores (อ่านว่า เซ-มา-ฟอร์) คือ "สัญญาณไฟจราจร" ของระบบปฏิบัติการครับ หน้าที่หลักของมันคือการควบคุมการเข้าถึงทรัพยากรที่มีอยู่อย่างจำกัด เพื่อไม่ให้ Process หลายตัวเข้าไปแย่งกันใช้งานพร้อมกันจนข้อมูลพัง

ในบริบทของ PostgreSQL ซึ่งมีการทำงานแบบหลายโปรเซส (Multi-process) Semaphores คือหัวใจสำคัญที่ใช้ประสานงานระหว่างโปรเซสต่างๆ ครับ

[^3]: System V Shared Memory (มักเรียกย่อว่า SysV SHM) คือกลไกการสื่อสารระหว่างโปรเซส (Inter-Process Communication - IPC) รูปแบบหนึ่งที่เก่าแก่และเสถียรที่สุดในระบบปฏิบัติการตระกูล Unix/Linux ครับ

[^4]: POSIX Shared Memory คือ มาตรฐานใหม่ (สอดคล้องกับมาตรฐาน POSIX) ที่ใช้ในการจองเนื้อที่หน่วยความจำส่วนกลางเพื่อให้ Process ต่างๆ เข้ามาใช้งานร่วมกันได้ โดยถูกพัฒนาขึ้นมาเพื่อแก้ข้อจำกัดและความยุ่งยากของ System V Shared Memory (แบบดั้งเดิม)
